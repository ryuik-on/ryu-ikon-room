# ROOM v2 rebuild plan

この工程はClaudeが計画・判断し、人間が重要事項を承認し、Codexが承認済みタスクを実装する前提です。一度に全体を完成させず、各Phaseの成果を確認してから次へ進みます。

## Phase 0 — 現状保全

- 目的: 公開中のv1を壊さず、いつでも復元できる状態にする
- 入力: `main`の確定HEAD、remote/Pages状態、v1ファイル
- 成果物: `room-v1-archive`タグ、`rebuild/room-v2`、`archive/room-v1/`、監査資料
- 確認項目: タグと`main`のcommit一致、archiveの同一性、remote未変更、復元手順
- 完了条件: v1がタグから復元でき、v2作業が独立ブランチだけで進められる

## Phase 1 — コンセプトと要件の確定

- 目的: 事実、原稿、方向性、未確定事項の境界を合意する
- 入力: `facts-and-content.md`、`visual-direction.md`、`open-questions.md`
- 成果物: 承認済み原稿、素材方針、優先順位、受け入れ条件
- 確認項目: チケット、料金、終演、主催表記、Founder’s Note、公式URL
- 完了条件: 公開に使う全情報が「確定」または「掲載しない」に分類される

## Phase 2 — 技術選定

- 目的: GitHub Pages、保守性、性能に合う最小構成を選ぶ
- 入力: コンテンツ量、演出要件、base path、運用者の習熟度
- 成果物: 技術選定案、比較、決定記録、開発/検証/公開手順
- 確認項目: Astro候補を含む静的生成方式、依存関係、更新方法、rollback
- 完了条件: 人間が構成と運用負担を承認し、コマンドが実環境で再現できる

## Phase 3 — 無装飾ワイヤーフレーム

- 目的: 色・写真・演出なしで情報順と導線を検証する
- 入力: `content-architecture.md`、確定原稿、モバイル要件
- 成果物: モバイル中心の無装飾ページ、主要幅のスクリーンショット
- 確認項目: 5秒認知、Ticket到達、見出し順、情報の重複、横はみ出し
- 完了条件: 内容構造と導線を人間が承認し、装飾なしで理解できる

## Phase 4 — Hero

- 目的: ROOM、日程、音楽公演であることを即時に伝える
- 入力: 承認済みHero素材、ロゴ、コピー、ワイヤーフレーム
- 成果物: Heroセクションと静止フォールバック
- 確認項目: 5秒認知、LCP、モバイルのトリミング、alt、reduced motion
- 完了条件: 主要幅と低速条件で情報がすぐ読め、素材権利も確認済み

## Phase 5 — 写真・映像

- 目的: 奏者と観客の間に生まれる空間を素材で伝える
- 入力: 採用素材、権利/クレジット、代替説明、容量方針
- 成果物: 画像/映像セクション、最適化素材、フォールバック
- 確認項目: 画角、読み込み、動画制御、音声、poster、alt、通信量
- 完了条件: モバイルで破綻せず、無効化/失敗時も意味が伝わる

## Phase 6 — Statement

- 目的: ROOMの理由を、承認済みの言葉で伝える
- 入力: コンセプト原文、Founder’s Note最終稿
- 成果物: Statement/Founder’s Noteセクション
- 確認項目: 原文一致、読みやすさ、改行、署名、写真との関係
- 完了条件: 人間が文章と表示を承認し、勝手な要約・統合がない

## Phase 7 — 公演情報

- 目的: 来場判断に必要な確定事実を明瞭に示す
- 入力: 確定した日時、会場、主催、料金
- 成果物: 公演情報セクション、必要なら会場導線
- 確認項目: 表記、曜日、時刻、料金区分、未確認情報の混入、構造化データとの一致
- 完了条件: 主催者確認済み情報だけが表示され、初見でも読み取れる

## Phase 8 — Ticket

- 目的: 有効な販売先へ迷わず移動できるようにする
- 入力: 確認済み販売URL、販売期間、注意事項
- 成果物: Ticket CTAと販売状態表示
- 確認項目: 到達先、外部遷移表示、キーボード操作、売切/未販売時、追跡方針
- 完了条件: 実リンクが正常で、誤購入につながる未確定情報がない

## Phase 9 — Footer・外部導線

- 目的: 主催者情報と継続接点を整える
- 入力: Instagram、公式サイト、主催者表記、必要なポリシー
- 成果物: SNS/公式サイト導線、Footer
- 確認項目: URL、リンク名、外部リンク、著作権表記、問い合わせ先
- 完了条件: すべての導線が確認済みで、不要なリンクがない

## Phase 10 — 演出

- 目的: コンセプト理解を助ける最小限の動きを加える
- 入力: 完成した静的ページ、演出仮説、性能予算
- 成果物: 段階的に検証可能な演出、motion-off代替
- 確認項目: 意味、量、スクロール拘束、入力方式、端末性能、reduced motion
- 完了条件: 演出なしでも成立し、演出が可読性・到達性・性能を損なわない

## Phase 11 — アクセシビリティ・パフォーマンス

- 目的: 多様な利用条件で安定して閲覧・操作できるようにする
- 入力: 全セクション、実素材、対象ブラウザ
- 成果物: 修正済み実装、測定結果、既知の制約
- 確認項目: キーボード、フォーカス、コントラスト、alt、ズーム、CWV候補、通信量
- 完了条件: 必須基準を満たし、残課題が公開可否判断に記録される

## Phase 12 — 公開前監査

- 目的: 内容、リンク、SEO、base path、復元手順を本番相当で確認する
- 入力: release candidate、`acceptance-criteria.md`、正式URL一覧
- 成果物: 監査記録、主要幅スクリーンショット、承認依頼
- 確認項目: build/lint、リンク、404、robots、sitemap、OGP、JSON-LD、秘密情報、差分
- 完了条件: 必須基準がすべて合格し、人間が切替を承認する

## Phase 13 — mainへの切り替え

- 目的: 承認済みv2を、rollback可能な形で公開する
- 入力: 承認済みrelease candidate、Pages設定、rollback手順
- 成果物: `main`への承認済み変更、公開確認、切替記録
- 確認項目: v1タグのremote存在、Pages source、公開URL、キャッシュ、監視、rollback
- 完了条件: 人間が本番表示を確認し、問題時にv1へ戻せる

## 各Phase共通ゲート

- 変更計画を先に提示する
- セクション単位で実装・確認する
- 主要幅のスクリーンショットを残す
- 未確認情報を追加しない
- build/lint/リンク確認を、その時点で採用済みの方法に合わせて実行する
- scope変更、公開、データ削除、外部連絡、有料操作は人間承認を得る
